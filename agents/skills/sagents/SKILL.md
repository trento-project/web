# sagents (v0.7.x)

Elixir framework for OTP-supervised AI agents with composable middleware, human-in-the-loop, scope-aware persistence, and Phoenix.PubSub-streamed state. Repo: https://github.com/sagents-ai/sagents/tree/v0.7.0

## When relevant

- Building / configuring `Sagents.Agent`, `AgentServer`, or `Coordinator`.
- Writing middleware (`Sagents.Middleware` behaviour) or extending built-ins.
- Wiring persistence behaviours (`AgentPersistence`, `DisplayMessagePersistence`, `FileSystemCallbacks`, `MessagePreprocessor`).
- Threading tenant scope through tools, sub-agents, callbacks.
- Subscribing LiveView / process to agent events.
- Migrating v0.6 → v0.7 (breaking: scope-first callback signatures).

For Trento-specific assistant integration (channel, tools, prompt) see the `trento-ai-assistant` skill instead.

## Scaffold vs primitives

Most examples in this bundle (`Coordinator`, `Factory`, `Conversations`, `IntegrationHelpers`, LiveView helpers) refer to modules generated by `mix sagents.setup` — the optional scaffold. They are the **common case** for an app adopting sagents fresh.

If your host app doesn't run the generator, you can still use sagents directly:

- `Sagents.Agent.new!/2` instead of a Factory
- `Sagents.AgentsDynamicSupervisor.start_agent_sync/1` instead of a Coordinator
- Your own state model instead of generated `Conversations` Ecto schemas

The patterns in these docs still apply; substitute your own module names. See the `trento-ai-assistant` skill for a worked example of direct-primitives integration (no scaffold).

## Cheat sheet

- AgentServer status: `:idle | :running | :interrupted | :paused | :cancelled | :error`.
- PubSub topic: `"agent_server:#{agent_id}"` (debug: `"agent_server:debug:#{agent_id}"`); events arrive as `{:agent, payload}`.
- `Agent.execute/3` → `{:ok, state}` | `{:interrupt, state, data}` | `{:pause, state}` | `{:error, reason}` | `{:ok, state, %ToolResult{}}` (when `until_tool` set).
- Middleware order: forward for `before_model`, reverse for `after_model`. **`HumanInTheLoop` must be last** in the list.
- `system_prompt/1` runs **once** at `Agent.new` — capture stable config in `init/1`, never embed user-controlled data.
- Scope is a **first-class field** on the Agent (`agent.scope`), passed as **first positional arg** to all persistence callbacks. Never stash in `state.metadata`.
- Metadata keys must be **strings** (JSONB serialization).
- `tool_info` payload in `:tool_call_identified` and `:tool_execution_update` events is **always atom-keyed** (`%{call_id:, name:, display_text:, arguments:, status: …}`). Never string keys. Verified at every construction site in `agent_server.ex` (`:2145, 2161, 2171, 2188, 2509, 2667, 2678, 2693, 2708, 2719`). Consumers can drop `tool_info["call_id"]` defensive fallbacks. `:call_id` is documented as always present in both `:tool_call_identified` and `:tool_execution_update :completed` per the moduledoc at `agent_server.ex:107-109`.
- `:status_changed, :cancelled, nil` only fires from `AgentServer.cancel/1` (single broadcast site at `agent_server.ex:1544`). If your host doesn't expose a cancel call, you can drop the `:cancelled` handler entirely — sagents won't emit unprompted. Sub-agent unwinding uses a different event family (`broadcast_fallback_subagent_cancel`), NOT `:status_changed, :cancelled`.
- **`agent.middleware` is `[%Sagents.MiddlewareEntry{module: mod, config: cfg}, ...]` post-`Agent.new`.** Test / inspection code must destructure via `& &1.module`, not `fn {mod, _} -> mod end`. Pre-init you pass `{mod, opts}` tuples; the framework wraps them.
- **`Sagents.AgentsDynamicSupervisor.start_agent_sync/1` spec is over-broad.** Declares `{:ok, pid()} | {:ok, pid(), :already_started} | {:error, term()}` but `wait_for_agent_ready/2` (the actual return path) only emits the 2-tuple. Adapter-pattern consumers can safely narrow their callback to `{:ok, pid()} | {:error, term()}` to match runtime reality.
- **`AgentServer.init/1` broadcasts initial `:idle`** at `agent_server.ex:1430`, BEFORE the first `:execute` call. Consumers subscribing immediately after `start_agent_sync` may receive a stale `:idle` (depending on race between `wait_for_agent_ready` and the init broadcast). Guard with a "we haven't seen `:running` yet for THIS run" flag if a stray `:idle` would confuse downstream UI.
- **If you nest an impl submodule whose name collides with `Sagents`** (e.g. `MyApp.Adapter.SagentsAgentServer`), `defdelegate ... to: Sagents.AgentServer` inside the nested module resolves relatively → `<parent>.Sagents.AgentServer` (undefined). Two ways out: (a) use `Elixir.Sagents.*` explicit prefix, or (b) keep impls in a separate namespace (e.g. `MyApp.Infrastructure.AI.SagentsAgentServer` — no collision, no prefix needed). Trento now uses (b).
- **`Sagents.AgentServer.subscribe(agent_id)` binds `self()`.** Adds the calling process to the PubSub topic. If your adapter call is inside a wrapper function called from a channel pid, the channel pid gets subscribed — usually what you want. Be deliberate about which process invokes it.
- **`:result` field on `:tool_execution_update :completed`** is the `inspect(tool_result)` string output of the tool's return value (set at `agent_server.ex` `on_tool_execution_completed` callback). Use directly when synthesizing AG-UI `TOOL_CALL_RESULT` content.
